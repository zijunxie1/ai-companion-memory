## 被审查的方案

### 项目背景（一段话）

Alice 是一款真人社交产品（类似 Soul/Tinder），AI 私聊是增加停留时长和 DAU 的附加功能。用户（PM）真实负责过 Memory 模块。当前在做求职作品集 Demo——V2 已完成真实 Memory 闭环（mem0 + Dify + Next.js + PostgreSQL），V3 要做 20 条 Eval Case + 评测面板 + LLM-as-Judge 自动评测。

### V3 评测方案核心主张

1. **时机判断**：系统跑通后、策略迭代前，必须先建评测集
2. **评测集规模**：20 条高质量 Case（核心 6-8 / 长尾对抗 6-8 / 安全 4-6）
3. **指标体系**：3 个强约束（False Memory Rate 接近零 / 删除 100% / Safety 一票否决）+ 4 个分档维度（召回准确率 / 不应召回率 / 回复自然度 / 连续性）
4. **评测方法**：LLM-as-Judge 自动评测（Dify Eval Judge Workflow）+ GSB 人工对比
5. **Baseline 先行**：先跑当前系统 Baseline，后续策略迭代时做 Before/After 对比
6. **面试叙事**：展示"从问题归因→Memory 设计→评测→Bad Case 回流"的完整闭环能力

### 评测集来源

- 人工构建（基于交接文档中定义的 Memory 问题类型和 5 个核心 Demo Case）
- 知识库方法论指导：按场景分（核心/长尾/安全）+ 按用户类型分（D3 回访/重度流失）

### 评测指标设计依据

- product.md 已定的强约束
- 知识库「评测指标与扣分规则设计方法」的分档标准
- 知识库「记忆系统四模块评估框架」的表示/提取/检索/维护维度
- 知识库「数据飞轮四阶段」的第一阶段定位

### 项目约束

- 这是求职作品集 Demo，不是生产系统
- 面试时间 1-2 个月后
- 面试官（外层用户）关注的是 PM 评测方法论，不是系统精度
- 评测结果需要可视化展示（评测面板）
- Demo 用户（内层用户）的 Memory 场景是深夜社交陪伴
