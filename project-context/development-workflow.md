# Vibe Coding 完整工作流

> 核心原则：不是把每一层全部做完再进入下一层，而是先识别最大的风险，再完成一条最小但真实、完整的端到端链路，并根据真实结果决定下一步。

---

## 1. 定义问题

### 目标

先定义要解决的问题，而不是先定义页面。

### 产出物

一页问题说明，至少包含：

- 目标用户；
- 核心使用场景；
- 用户当前如何解决；
- 现有方案为什么不够；
- 成功指标；
- 明确不做的事情；
- 最危险的三个假设。

### 进入下一步的门

能够回答：

> 谁会在什么场景下使用它，使用之后什么结果会发生变化？

---

## 2. 对标调研与风险排序

### 对标调研

同步检查：

- 同类产品；
- GitHub 开源项目；
- 常见页面和交互结构；
- 开源许可证；
- 项目维护状态；
- Issue 积压；
- 技术栈与当前项目是否匹配；
- 是否存在明显安全或依赖风险。

对标的目的是减少重复造轮子，不是照抄视觉。

### 风险分类

从以下五类风险中找出最致命的 1—3 个：

1. 需求风险：用户是否真的需要；
2. 技术风险：模型、接口或性能是否可行；
3. 数据风险：数据是否能获得、是否可靠；
4. 安全风险：权限、隐私和数据隔离是否可控；
5. 商业风险：成本、定价和规模是否成立。

### 产出物

- 对标与开源方案表；
- 排序后的 Top 1—3 风险清单。

### 进入下一步的门

能够说清楚：

> 这个项目最可能因为什么失败？

---

## 3. 验证最高风险

根据风险选择验证方式：

| 不确定项 | 优先验证方式 |
|---|---|
| 技术是否可行 | 使用真实数据和真实模型做 Spike |
| 交互是否合理 | 快速静态 Demo |
| 需求是否成立 | 目标用户访谈、人工代办或真实任务测试 |
| 成本是否可承受 | 单次成本和用户规模测算 |
| 权限是否可控 | 最小权限模型和越权测试原型 |

### 验证前必须写阈值

每次验证至少明确：

- 验证什么；
- 使用什么样本；
- 什么结果算通过；
- 什么结果需要调整；
- 什么结果意味着停止或换方案；
- 最多投入多少时间或成本。

### 静态 Demo 的使用边界

静态 Demo 用于验证：

- 页面是否看得懂；
- 信息架构是否合理；
- 操作入口是否明确；
- 交互方向是否符合预期。

静态 Demo 不能证明：

- 用户真的存在该问题；
- 用户愿意改变工作习惯；
- 用户愿意付费；
- 技术和数据链路真实可行。

规则：

- 时间控制在几十分钟到半天；
- 每次只验证一个明确问题；
- 默认可丢弃；
- 不过度精修；
- 不把“页面不错”误判为“需求已验证”。

### 进入下一步的门

最高风险达到预先定义的通过阈值，或已形成明确的调整方案。

---

## 4. 定义第一条纵向切片

不要定义第一版需要多少页面，而要定义一条最小闭环：

```text
用户输入
→ 系统处理
→ 数据保存
→ 用户获得结果
```

### 同步形成的草图级材料

- 简化 PRD；
- 1—2 个核心页面稿；
- 核心实体和关系；
- 核心 API 请求与响应；
- 权限矩阵；
- 可客观判断的验收标准。

数据模型、权限和接口必须在设计阶段同步考虑，因为它们会直接影响交互。

### 进入下一步的门

能够明确回答：

- 第一条切片包含什么；
- 不包含什么；
- 什么结果算完成。

---

## 5. 建仓库与工程基线

### 最小工程基线

所有项目至少具备：

- 一条命令可以启动；
- 依赖版本锁定；
- 密钥不会提交到仓库；
- Lint 和类型检查可运行；
- README 说明启动方式；
- 使用 Git 管理版本。

### 按风险提高基线

#### 本地工具或展示原型

- 最小工程基线即可。

#### 有账号和云数据库

增加：

- CI；
- 基础测试；
- 日志；
- 环境变量分环境管理；
- 基础权限检查。

#### 涉及支付、隐私文件或企业数据

增加：

- 权限负例测试；
- 用户和组织数据隔离；
- 备份；
- 成本与调用限额；
- 审计或关键操作记录。

#### 高风险或不可逆操作

增加：

- 更完整测试；
- 安全审查；
- 人工确认；
- 回滚机制；
- 发布后验证。

### 正式产品的前端骨架

正式产品可进一步建立：

- 设计风格；
- 技术方案；
- 目录边界；
- 样式系统；
- 组件规范；
- 实施计划。

组件抽象原则：

> 只有当多个组件具有相同语义、相同交互逻辑和稳定的共同变化方向时才抽象；不要因为外观相似或仅出现两次就强行抽象。

### AI 项目上下文

建议建立：

```text
project-context/
├── product.md
├── architecture.md
├── data-model.md
├── api-contracts.md
├── permissions.md
├── conventions.md
└── decisions/
```

未实现的计划不能提前写入正式契约，必须先进入任务草案。

---

## 6. 完成第一条真实链路

第一条切片必须尽量包含：

```text
真实页面
+ 真实 API
+ 真实数据库
+ 必要权限
+ 基础错误处理
```

### 工作方式

- 每个功能使用独立分支；
- 新 AI 会话是可选项，不是必要条件；
- 将任务拆到可以人工 Review 的粒度；
- 避免无人值守的长链自动迭代；
- 不做与当前切片无关的重构。

### 最低测试护栏

至少保留：

1. 一条核心成功流程；
2. 一条关键失败或越权流程；
3. 一个最重要的业务规则。

### 进入下一步的门

用户本人使用真实数据能够完成核心任务，且结果可以用于下一步。

---

## 7. 内部验收与轻量外部试用

### 内部验收

用户本人：

- 使用真实数据；
- 完整走通核心流程；
- 确认输出有价值；
- 记录卡点、冗余步骤和错误状态。

### 轻量外部试用

准备让其他人使用前，至少让少量非开发者独立尝试。

观察：

- 是否能完成核心任务；
- 产出是否被真实采用；
- 哪一步需要额外解释；
- 哪一步阻塞或多余。

不要只问“喜不喜欢”。

### 进入下一步的门

核心任务可完成，且产出被实际用于后续动作。

---

## 8. 按纵向切片持续扩展

每个新功能使用以下循环：

```text
Issue / 需求
→ 验收标准
→ UI、数据、接口、权限和迁移影响分析
→ 必要时先做 Spike
→ 创建独立分支
→ Builder 实现完整纵向切片
→ 自动测试和人工 Review
→ 代码、测试、迁移和正式契约在同一 PR 合并
→ 部署和验证
```

### 任务说明至少包含

- 目标；
- 非目标；
- 验收标准；
- 允许修改范围；
- 禁止修改范围；
- 必须复用的公共能力；
- 必须运行的测试；
- Change Request 条件。

### Git 原则

- 一个功能一个分支；
- 一个 PR 只解决一个主要问题；
- PR 内可以有多个逻辑清楚的 commit；
- 多个 Agent 真正并行时再使用 Worktree。

---

## 9. 随用户数量和风险逐步加固

### 原型阶段

- 不保存不必要的敏感数据；
- 保护密钥；
- 有基础错误处理。

### 内测阶段

- 基础权限；
- 错误日志；
- 模型调用和成本记录；
- 核心回归测试。

### 小范围上线

- 备份；
- 调用限额；
- 健康检查；
- 一键回滚；
- 基础告警。

### 正式产品

- 恢复演练；
- 完整权限检查；
- 持续监控；
- 数据迁移策略；
- 更严格的安全审查。

安全不是最后统一检查，而是：

- 设计阶段定义数据所有权和权限；
- 开发阶段执行密钥、输入和越权检查；
- 发布前完成生产风险和回滚检查。

---

## 10. 用真实数据决定下一步

上线后持续检查：

- 错误和崩溃；
- 核心任务完成率；
- 用户在哪一步退出；
- AI 输出是否被采用；
- 单用户成本；
- 是否存在滥用或异常调用；
- 哪些功能长期无人使用。

根据证据选择：

- 继续；
- 调整；
- 删除；
- 停止。

> 上线不是终点，而是第一次获得真实运行证据的开始。

---

## 反模式速查

| 反模式 | 正确做法 |
|---|---|
| 一个会话从设计聊到部署 | 短任务、文件对齐、按阶段交接 |
| 做完所有页面再做后端 | 先完成一条真实纵向切片 |
| 把聊天历史当项目文档 | 重要事实写入仓库 |
| 巨型 Prompt 一次执行全部工作 | 先计划，再按可 Review 粒度执行 |
| 盲目信任 AI 输出 | 独立 Review 和自动测试 |
| 一个 PR 修改所有内容 | 一个 PR 解决一个主要问题 |
| 最后才考虑安全 | 按风险和阶段逐步加固 |
| 验证不设阈值 | 提前定义通过、调整和停止标准 |
| 未验证骨架就不断加功能 | 工程基线达到当前风险要求后再继续 |
| 无人值守长链循环 | 人类审批关键门，Agent 按任务执行 |
