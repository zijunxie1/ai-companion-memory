## 核心问题

### 问题归因

Alice 用户流失按生命周期分层：

| 流失层级 | 占比 | Memory 是否主责 | 本项目覆盖 |
|---|---|---|---|
| 首日流失 | ~50% | ❌ 否（首屏/角色/首聊） | 不覆盖 |
| 短期流失（D3/D7） | ~22% | ⚠️ 部分 | **覆盖** |
| 中期流失（3-7天） | ~13% | ✅ 是 | **核心** |
| 重度流失（断崖） | ~8% | ✅ 最高优先级 | **核心** |
| 高价值未转化 | ~7% | ⚠️ 部分 | 后续 |

### 问题陈述

在 Alice 这类 AI 陪伴社交产品中，D3/D7 之后的用户流失很大程度上源于 AI"没有记忆"或"记错了"——表现为失忆、误记、重复、剧情断裂、人设不一致和记忆不可控。本项目通过构建一套 Memory 驱动的关系连续性系统，解决中期和重度用户因 Memory 问题导致的流失。

## 产品目标

1. **D3/D7 回访连续感**：用户回来时 AI 记得上次话题、用户偏好、当前关系阶段
2. **降低中期重复与关系停滞**：共同经历、关系成长、非模板化回复
3. **减少重度用户断崖流失**：关键记忆保护、Persona 一致性、Bad Case 回归
4. **（后续）增强长期价值和付费感知**

---

## 非目标

1. 不做独立 App——是 Alice 内的 AI 私聊功能
2. 不解决首日流失——由 Leader 负责
3. 不做生产级系统——是产品级验证 Demo
4. 第一版不做完整主动聊天系统——保留演示 Case
5. 第一版不做完整 Safety OS——Safety 作为底线存在
6. 不做多角色共享 Memory / 多模态 Memory / 复杂剧情引擎
7. 不做自行训练模型
8. 本项目阶段不触碰真实公司敏感数据

## 产品形态

P1 不是独立 App，是现有社交产品中的新增 AI 私聊能力。

- AI 角色出现在消息列表
- AI 消息和真人消息共存在同一消息体系
- 用户进入 AI 私聊页
- AI 受原产品用户状态、通知、免打扰和关系规则约束
- 用户界面像真实社交产品的私聊功能，不是 ChatGPT，不直接展示 Dify

### 三层 Demo

1. **用户侧聊天界面**：消息列表、AI 私聊页、AI 角色主页、Memory 管理页、主动消息设置、隐私安全设置
2. **产品策略后台**：用户画像、关系阶段、Memory、Prompt 版本、Persona、主动触发规则、安全规则
3. **Eval、Trace 和 Bad Case 面板**：Memory 使用、Prompt 版本、工作流节点、安全策略、延迟、Token、Eval 分数、Bad Case

---

## 第一版 MVP 范围

聚焦四项 Memory 能力：

1. 写入（什么值得记）
2. 召回（当前场景该用哪些 Memory）
3. 更新和冲突（纠正后旧 Memory 不再生效）
4. 删除（删除后不再被召回）

Safety 作为底线存在，不做完整体系。关系阶段作为 Memory 召回变量。主动聊天保留一个演示 Case。

## 成功指标

### Demo 阶段可展示成功标准

1. 面试官可在 Demo 中完成一条真实聊天链路
2. 能展示 Before / After 对比
3. 能展示 Memory 写入、召回、更新、删除完整生命周期
4. 能展示 Trace
5. 20 条离线 Eval Case 有明确结果
6. 有 Bad Case 复盘

### Demo 阶段可量化成功标准（离线 Eval）

- 删除 Case 必须 100% 通过
- Safety Case 一票否决
- False Memory Rate 接近零
- 具体阈值待 Baseline 后确定

## Dify 的定位

Dify 是 AI 编排中台和快速验证层，不是产品前台。

业务状态、用户控制权、Memory 数据、Trace 和 Eval 沉淀在自有系统；模型编排和生成链路放在 Dify。
